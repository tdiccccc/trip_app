# E2E テストシナリオ: 003 しおり（タイムライン）管理

## 概要

| 項目 | 内容 |
|------|------|
| シナリオID | 003 |
| 機能 | しおり（タイムライン）管理 |
| 対象ページ | `/trips/:tripId/itinerary` |
| 関連API | `GET /api/trips/{tripId}/itinerary?date={date}`, `POST /api/trips/{tripId}/itinerary`, `PATCH /api/trips/{tripId}/itinerary/{id}`, `DELETE /api/trips/{tripId}/itinerary/{id}`, `PATCH /api/trips/{tripId}/itinerary/reorder` |
| 前提条件 | アプリ起動済み、ログイン済み、旅行が1件以上存在（tripId=1、日程: 複数日） |
| テストユーザー | たろう (`taro@example.com` / `password`) |

---

## 前提条件

- フロントエンド（Nuxt）が `http://localhost:3900` で起動していること
- バックエンド（Laravel）が `http://localhost:8900` で起動していること
- `php artisan migrate --seed` が実行済みで、テストユーザーおよび旅行データが存在すること
- tripId=1 の旅行が複数日の日程を持つこと（start_date と end_date が異なる日付）
- ログイン済みの状態で開始すること（TC-001-02 を事前に実行済み）

---

## テストケース一覧

| # | ケース名 | 種別 | 概要 |
|---|---------|------|------|
| TC-003-01 | しおりページの表示 | 正常系 | しおりページが正しく表示され、日付タブ・切替ボタン・追加ボタンが存在する |
| TC-003-02 | 行程の追加 | 正常系 | TimelineForm モーダルから新しい行程を追加し、タイムラインに表示される |
| TC-003-03 | 行程の編集 | 正常系 | 既存の行程を編集し、変更が反映される |
| TC-003-04 | 行程の削除 | 正常系 | 既存の行程を削除し、リストから消える |
| TC-003-05 | 日付タブの切り替え | 正常系 | 日付タブを切り替えるとその日の行程が表示される |
| TC-003-06 | マップビューの切り替え | 正常系 | リスト/マップの表示切替が正しく動作する |

## TC-003-01: しおりページの表示

### 目的

しおりページにアクセスし、日付タブ・リスト/マップ切替ボタン・追加ボタン（FAB）が全て表示されることを確認する。

### 手順

```
Step 1: しおりページに遷移する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/trips/1/itinerary" }
  期待結果: ページが正常にロードされる

Step 2: ページのスナップショットを取得して UI 要素を確認する
  Tool: browser_snapshot
  期待結果: 以下の要素が存在する
    - 日付タブが旅行日程分（start_date ~ end_date）表示されている
    - 1日目のタブが選択状態になっている
    - 「リスト」「マップ」切替ボタンが表示されている
    - 追加ボタン（FAB、+アイコン）が表示されている
    - default レイアウト（ヘッダー・ナビあり）で表示されている

Step 3: 既存の行程が表示されていることを確認する
  Tool: browser_snapshot
  期待結果:
    - Seeder で投入済みの行程がタイムライン形式で表示されている
    - 各行程にタイトル・時刻・交通手段アイコンが表示されている
    ※ 行程が0件の場合は空状態（「行程がありません」等）が表示される
```

### 判定基準

- URL が `/trips/1/itinerary` であること
- 日付タブが旅行の日程数分表示されていること
- リスト/マップ切替ボタンが存在すること
- 追加ボタン（FAB）が存在すること
- 既存の行程またはから状態メッセージが表示されていること

## TC-003-03: 行程の編集

### 目的

### 前提

### 手順

```
Step 1: しおりページに遷移する

  Params: { "url": "http://localhost:3900/trips/1/itinerary" }
  期待結果: しおりページが表示される

Step 2: スナップショットを取得して既存の行程を確認する
  Tool: browser_snapshot
  期待結果: 「伊勢神宮 内宮参拝」の行程が表示されている

Step 3: 行程の編集ボタンをクリックする
  Tool: browser_click
  Params: { "element": "「伊勢神宮 内宮参拝」の編集ボタン" }
  ※ スナップショットの ref 番号を使用する
  期待結果: TimelineForm モーダルが開き、既存の値がセットされた状態で表示される

Step 4: 編集フォームに既存値がセットされていることを確認する
  Tool: browser_snapshot
  期待結果:
    - タイトルフィールドに「伊勢神宮 内宮参拝」がセットされている

    - 終了時刻に「11:00」がセットされている
    - メモに「早朝がおすすめ」がセットされている
    - 交通手段「電車」が選択状態になっている

  期待結果: タイトルフィールドが「伊勢神宮 外宮参拝」に変更される

Step 6: 保存ボタンをクリックする
  Tool: browser_click
  Params: { "element": "保存ボタン" }
```

- タイトルの変更が保存されること（`PATCH /api/trips/{tripId}/itinerary/{id}` が成功）
- 変更した内容がタイムライン上に正しく反映されること
- 変更していないフィールド（時刻・メモ・交通手段）は元の値が維持されること

## 実行時の注意事項

### テスト実行順序

テストケースは番号順に実行することを推奨する。特に以下の依存関係に注意する:

1. TC-003-01 は独立して実行可能（ログイン済みであること）
2. TC-003-02 は行程を追加するため、TC-003-03 / TC-003-04 の前提になる
3. TC-003-03 は TC-003-02 で追加した行程を編集する
4. TC-003-04 は TC-003-03 で編集した行程を削除する
5. TC-003-05 / TC-003-06 は独立して実行可能（ログイン済みであること）

### 推奨実行順

1. TC-003-01（ページ表示確認）
2. TC-003-05（日付タブ切り替え -- 既存データで確認）
3. TC-003-06（マップビュー切り替え -- 既存データで確認）
4. TC-003-02（行程の追加）
5. TC-003-03（行程の編集 -- TC-003-02 で追加した行程を使用）
6. TC-003-04（行程の削除 -- TC-003-03 で編集した行程を使用）

### ログイン前提

全テストケースはログイン済みの状態で実行する。未認証の場合は `/login` にリダイレクトされるため、事前に TC-001-02（正常ログイン）を実行しておくこと。

### browser_snapshot の ref 番号について

- `browser_snapshot` で取得したスナップショットには、各インタラクティブ要素に ref 番号が割り当てられる
- `browser_click` や `browser_type` では、この ref 番号を使って要素を指定する
- 本仕様書では要素の説明テキストで記載しているが、実行時はスナップショットの ref 番号に置き換えること

### テキスト入力の上書きについて

- TC-003-03（編集）でタイトルを変更する際は、既存のテキストを全選択してから上書きする必要がある
- `browser_type` の `clear` パラメータを使用するか、Ctrl+A で全選択してから入力する

### 削除確認ダイアログについて

- TC-003-04 の削除確認は、実装によりブラウザネイティブの `confirm()` ダイアログまたはカスタムモーダルが使われる可能性がある
- ブラウザネイティブの場合は `browser_handle_dialog` 等の適切な MCP ツールで対応する
- 実行時にスナップショットで確認し、適切な操作を選択すること

### エラー発生時の対応

- MCP ツールの呼び出しでエラーが返された場合は、即座にエラー内容を報告する
- 回避策の実行やリトライは行わない（CLAUDE.md の Playwright MCP 使用ルールに準拠）
- 期待結果と異なる場合は、実際の状態をスクリーンショットまたはスナップショットで記録し、差異を報告する
