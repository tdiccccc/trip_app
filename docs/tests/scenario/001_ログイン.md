# E2E テストシナリオ: 001 ログイン

## 概要

| 項目 | 内容 |
|------|------|
| シナリオID | 001 |
| 機能 | ログイン（Sanctum SPA 認証） |
| 対象ページ | `/login` |
| 関連API | `GET /sanctum/csrf-cookie`, `POST /api/login`, `GET /api/user` |
| 前提条件 | アプリが起動済み（フロントエンド: port 3900、バックエンド: port 8900） |
| テストユーザー | たろう (`taro@example.com` / `password`)、はなこ (`hanako@example.com` / `password`) |

---

## 前提条件

- フロントエンド（Nuxt）が `http://localhost:3900` で起動していること
- バックエンド（Laravel）が `http://localhost:8900` で起動していること
- `php artisan migrate --seed` が実行済みで、テストユーザーが存在すること
- ブラウザの Cookie / セッションがクリアされた状態で開始すること

---

## テストケース一覧

| # | ケース名 | 種別 | 概要 |
|---|---------|------|------|
| TC-001-01 | ログインページの表示 | 正常系 | ログインページが正しく表示される |
| TC-001-02 | 正常ログイン（たろう） | 正常系 | 正しい認証情報でログインし、旅行一覧にリダイレクトされる |
| TC-001-03 | 正常ログイン（はなこ） | 正常系 | 2人目のユーザーでもログインできる |
| TC-001-04 | 不正なメールアドレス | 異常系 | 存在しないメールアドレスでエラーが表示される |
| TC-001-05 | 不正なパスワード | 異常系 | 正しいメールアドレスに誤ったパスワードでエラーが表示される |
| TC-001-06 | 空欄送信 | 異常系 | 未入力のままログインボタンを押してもバリデーションで弾かれる |
| TC-001-07 | 認証済みユーザーのリダイレクト | 正常系 | ログイン済み状態で `/login` にアクセスすると `/trips` にリダイレクトされる |
| TC-001-08 | 未認証ユーザーの認証ガード | 正常系 | 未認証で認証必須ページにアクセスすると `/login` にリダイレクトされる |
| TC-001-09 | ログアウト | 正常系 | ログアウト後にログインページに遷移し、認証状態がクリアされる |

## TC-001-01: ログインページの表示

### 目的

ログインページが正しくレンダリングされ、必要な UI 要素が全て表示されることを確認する。

### 手順

```
Step 1: ログインページに遷移する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/login" }
  期待結果: ページが正常にロードされる

Step 2: ページのスナップショットを取得して UI 要素を確認する
  Tool: browser_snapshot
  期待結果: 以下の要素が存在する
    - タイトル「Ise Trip」（h1 要素）
    - サブテキスト「ログインして続けましょう」
    - 「メールアドレス」ラベルとメール入力フィールド（id="email"）
    - 「パスワード」ラベルとパスワード入力フィールド（id="password"）
    - 「ログイン」ボタン（type="submit"）
    - エラーメッセージは表示されていない（role="alert" の要素がない）
```

### 判定基準

- ページタイトルに「ログイン」が含まれる
- フォーム要素（メール入力、パスワード入力、送信ボタン）が全て存在する
- blank レイアウト（ヘッダー・ナビなし）で表示されている

## TC-001-02: 正常ログイン（たろう）

### 目的

正しい認証情報でログインし、旅行一覧ページ（`/trips`）にリダイレクトされることを確認する。

### 手順

```
Step 1: ログインページに遷移する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/login" }
  期待結果: ログインページが表示される

Step 2: スナップショットを取得してフォーム要素を確認する
  Tool: browser_snapshot
  期待結果: メールアドレス入力フィールドとパスワード入力フィールドが確認できる

Step 3: メールアドレスを入力する
  Tool: browser_click
  Params: { "element": "メールアドレス入力フィールド（id='email'）" }
  ※ スナップショットの ref 番号を使用する

  Tool: browser_type
  Params: { "text": "taro@example.com" }
  期待結果: メールアドレスフィールドに値が入力される

Step 4: パスワードを入力する
  Tool: browser_click
  Params: { "element": "パスワード入力フィールド（id='password'）" }
  ※ スナップショットの ref 番号を使用する

  Tool: browser_type
  Params: { "text": "password" }
  期待結果: パスワードフィールドに値が入力される

Step 5: ログインボタンをクリックする
  Tool: browser_click
  Params: { "element": "ログインボタン" }
  ※ スナップショットの ref 番号を使用する
  期待結果: ログイン処理が開始される（ボタンが「ログイン中...」に変わる）

Step 6: リダイレクト完了を待ってスナップショットを取得する
  Tool: browser_snapshot
  期待結果:
    - URL が /trips を含む（旅行一覧ページにリダイレクトされている）
    - ログインフォームは表示されていない
    - 旅行一覧ページのコンテンツが表示されている
    
  ※ リダイレクトが完了していない場合は数秒待ってから再度スナップショットを取得する
```

### 判定基準

- ログイン後に `/trips` ページにリダイレクトされる
- エラーメッセージが表示されない
- 旅行一覧ページが正しく表示される（default レイアウトに切り替わる）

## TC-001-03: 正常ログイン（はなこ）

### 目的

2人目のユーザー（はなこ）でもログインできることを確認する。

### 手順

```
Step 1: Cookie をクリアするためにログアウト状態にする
  ※ 前のテストでログイン済みの場合は、先にログアウトするか、
     新しいブラウザコンテキストで実行する

  期待結果: ログインページが表示される

Step 2: スナップショットを取得する
  Tool: browser_snapshot
  期待結果: ログインフォームが表示される
  ※ もし /trips にリダイレクトされた場合は、前テストのセッションが残っている。
     その場合は先にログアウト操作（TC-001-09 参照）を行ってから再実行する。

Step 3: メールアドレスを入力する
  Tool: browser_click
  Params: { "element": "メールアドレス入力フィールド" }
  Tool: browser_type
  Params: { "text": "hanako@example.com" }

Step 4: パスワードを入力する
  Tool: browser_click
  Params: { "element": "パスワード入力フィールド" }
  Tool: browser_type
  Params: { "text": "password" }

Step 5: ログインボタンをクリックする
  Tool: browser_click
  Params: { "element": "ログインボタン" }

Step 6: リダイレクト完了を確認する
  Tool: browser_snapshot
  期待結果:
    - URL が /trips を含む
    - 旅行一覧ページが表示される
```

### 判定基準

- はなこのアカウントでログインが成功する
- `/trips` にリダイレクトされる

## TC-001-06: 空欄送信

### 目的

メールアドレスとパスワードを未入力のままログインボタンを押した場合、ブラウザのバリデーションで送信が阻止されることを確認する。

### 手順

```
Step 1: ログインページに遷移する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/login" }

Step 2: スナップショットを取得する
  Tool: browser_snapshot
  期待結果: フォームが空の状態で表示される

Step 3: 何も入力せずにログインボタンをクリックする
  Tool: browser_click
  Params: { "element": "ログインボタン" }

Step 4: バリデーション結果を確認する
  Tool: browser_snapshot
  期待結果:
    - URL は /login のまま
    - ブラウザネイティブのバリデーションにより送信が阻止される
      （input に required 属性が設定されているため）
    - API リクエストは送信されない

---

追加ケース: メールアドレスのみ入力してパスワード空欄

Step 5: メールアドレスのみ入力する
  Tool: browser_click
  Params: { "element": "メールアドレス入力フィールド" }
  Tool: browser_type
  Params: { "text": "taro@example.com" }

Step 6: ログインボタンをクリックする
  Tool: browser_click
  Params: { "element": "ログインボタン" }

Step 7: バリデーション結果を確認する
  Tool: browser_snapshot
  期待結果:
    - URL は /login のまま
    - パスワードフィールドの required バリデーションにより送信が阻止される
```

### 判定基準

- フォーム送信が阻止される（HTML5 required バリデーション）
- API リクエストが発生しない
- ページ遷移が発生しない

## TC-001-07: 認証済みユーザーのリダイレクト

### 目的

既にログイン済みのユーザーが `/login` にアクセスした場合、自動的に `/trips` にリダイレクトされることを確認する。

### 前提

TC-001-02 が成功した直後の状態（ログイン済み）で実行する。

### 手順

```
Step 1: まずログインする（TC-001-02 の Step 1-5 を実行）
  ※ 既にログイン済みの場合はスキップ

Step 2: ログインページに直接遷移する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/login" }

Step 3: リダイレクト結果を確認する
  Tool: browser_snapshot
  期待結果:
    - URL が /trips を含む（/login ではない）
    - 旅行一覧ページが表示される
    - ログインフォームは表示されない
```

### 判定基準

- ログイン済みユーザーは `/login` にアクセスできない
- 自動的に `/trips` にリダイレクトされる

## TC-001-09: ログアウト

### 目的

ログアウト操作により認証状態がクリアされ、ログインページに遷移することを確認する。

### 前提

ログイン済みの状態で開始する。

### 手順

```
Step 1: ログインする（TC-001-02 の手順を実行）
  ※ 既にログイン済みの場合はスキップ

Step 2: ログイン後のページでスナップショットを取得する
  Tool: browser_snapshot
  期待結果:
    - 旅行一覧ページが表示されている
    - ヘッダーまたはナビゲーションにログアウト導線が存在する

Step 3: ログアウトボタン/リンクを探してクリックする
  Tool: browser_click
  Params: { "element": "ログアウトボタンまたはリンク" }
  ※ スナップショットで確認した ref 番号を使用する
  ※ ログアウト導線の配置はヘッダーまたはナビゲーション内にある
  期待結果: ログアウト処理が実行される

Step 4: ログアウト後の状態を確認する
  Tool: browser_snapshot
  期待結果:
    - URL が /login を含む
    - ログインフォームが表示される

Step 5: 認証状態がクリアされたことを確認する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/trips" }

Step 6: 認証ガードの動作を確認する
  Tool: browser_snapshot
  期待結果:
    - URL が /login を含む（認証が切れているためリダイレクトされる）
    - ログインフォームが表示される
```

### 判定基準

- ログアウト後に `/login` にリダイレクトされる
- セッションが無効化されている（認証必須ページにアクセスできない）
- `POST /api/logout` が呼ばれ、サーバー側のセッションも破棄されている

## 実行時の注意事項

### テスト実行順序

テストケースは番号順に実行することを推奨する。特に以下の依存関係に注意する:

1. TC-001-01 は独立して実行可能
2. TC-001-02 はログイン後の状態を作るため、TC-001-07 の前提になる
3. TC-001-07 はログイン済み状態が必要
4. TC-001-08 は未認証状態が必要（TC-001-09 の後に実行するか、新しいコンテキストで実行）
5. TC-001-09 はログイン済み状態が必要

### 推奨実行順

1. TC-001-01（ページ表示確認）
2. TC-001-04（異常系: 不正メール）
3. TC-001-05（異常系: 不正パスワード）
4. TC-001-06（異常系: 空欄送信）
5. TC-001-08（認証ガード: 未認証状態で実施）
6. TC-001-02（正常ログイン: たろう）
7. TC-001-07（認証済みリダイレクト: ログイン直後に実施）
8. TC-001-09（ログアウト）
9. TC-001-03（正常ログイン: はなこ -- ログアウト直後に実施）

### browser_snapshot の ref 番号について

- `browser_snapshot` で取得したスナップショットには、各インタラクティブ要素に ref 番号が割り当てられる
- `browser_click` や `browser_type` では、この ref 番号を使って要素を指定する
- 本仕様書では要素の説明テキストで記載しているが、実行時はスナップショットの ref 番号に置き換えること

### エラー発生時の対応

- MCP ツールの呼び出しでエラーが返された場合は、即座にエラー内容を報告する
- 回避策の実行やリトライは行わない（CLAUDE.md の Playwright MCP 使用ルールに準拠）
- 期待結果と異なる場合は、実際の状態をスクリーンショットまたはスナップショットで記録し、差異を報告する
