# E2E テストシナリオ: 002 旅行管理

## 概要

| 項目 | 内容 |
|------|------|
| シナリオID | 002 |
| 機能 | 旅行管理（作成・表示・編集・削除） |
| 対象ページ | `/trips`, `/trips/new`, `/trips/:tripId` |
| 関連API | `GET /api/trips`, `POST /api/trips`, `GET /api/trips/{tripId}`, `PATCH /api/trips/{tripId}`, `DELETE /api/trips/{tripId}`, `GET /api/users` |
| 前提条件 | アプリが起動済み（フロントエンド: port 3900、バックエンド: port 8900）、ログイン済み |
| テストユーザー | たろう (`taro@example.com` / `password`)、はなこ (`hanako@example.com` / `password`) |

---

## 前提条件

- フロントエンド（Nuxt）が `http://localhost:3900` で起動していること
- バックエンド（Laravel）が `http://localhost:8900` で起動していること
- `php artisan migrate --seed` が実行済みで、テストユーザーが存在すること
- たろう (`taro@example.com` / `password`) でログイン済みの状態で開始すること
- TC-002-07 では、はなこ (`hanako@example.com` / `password`) もメンバー候補として存在すること

---

## テストケース一覧

| # | ケース名 | 種別 | 概要 |
|---|---------|------|------|
| TC-002-01 | 旅行一覧ページの表示 | 正常系 | 旅行一覧ページが正しく表示される |
| TC-002-02 | 旅行作成ページの表示 | 正常系 | 旅行作成フォームが正しく表示される |
| TC-002-03 | 旅行の新規作成 | 正常系 | フォーム入力後に旅行を作成し、旅行トップにリダイレクトされる |
| TC-002-04 | 旅行トップページの表示 | 正常系 | 旅行詳細情報とクイックリンクが正しく表示される |
| TC-002-05 | 旅行情報の編集（ownerのみ） | 正常系 | 編集モーダルからタイトルを変更して画面に反映される |
| TC-002-06 | 旅行の削除（ownerのみ） | 正常系 | 旅行を削除して一覧から消えることを確認する |
| TC-002-07 | メンバー追加した旅行作成 | 正常系 | メンバーを選択して旅行を作成し、メンバーが表示される |

## TC-002-01: 旅行一覧ページの表示

### 目的

旅行一覧ページが正しくレンダリングされ、ヘッダーと「新しい旅行」ボタンが表示されることを確認する。

### 手順

```
Step 1: 旅行一覧ページに遷移する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/trips" }
  期待結果: ページが正常にロードされる

Step 2: ページのスナップショットを取得して UI 要素を確認する
  Tool: browser_snapshot
  期待結果: 以下の要素が存在する
    - ページタイトル「旅行一覧」（h1 要素）
    - 「新しい旅行」リンクボタン（/trips/new へのリンク）
    - 旅行カードが表示されている、または「まだ旅行がありません」の空状態が表示される
    - default レイアウト（ヘッダー・ナビ付き）で表示されている
```

### 判定基準

- ページタイトルに「旅行一覧」が含まれる
- 「新しい旅行」ボタンが存在し、`/trips/new` へのリンクになっている
- default レイアウトで表示されている（ヘッダー・ナビが存在する）

## TC-002-02: 旅行作成ページの表示

### 目的

### 手順


## TC-002-06: 旅行の削除（ownerのみ）

### 目的

旅行の owner が旅行を削除し、確認ダイアログで承認後に旅行が削除され、一覧ページにリダイレクトされることを確認する。

### 前提

TC-002-05 で編集した旅行（または TC-002-03 で作成した旅行）が存在する状態で実行する。

### 手順

```
Step 1: 旅行トップページに遷移する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/trips/{tripId}" }
  ※ {tripId} は削除対象の旅行 ID に置き換える
  期待結果: 旅行トップページが表示される

Step 2: スナップショットを取得して削除ボタンを確認する
  Tool: browser_snapshot
  期待結果: ページ下部に「この旅行を削除」ボタンが表示される

Step 3: 削除ボタンをクリックする
  Tool: browser_click
  Params: { "element": "この旅行を削除ボタン" }
  ※ スナップショットの ref 番号を使用する
  期待結果: ブラウザの confirm ダイアログが表示される
    - メッセージ: 「この旅行を削除しますか？紐づく全てのデータが削除されます。」

Step 4: 確認ダイアログで OK をクリックする
  Tool: browser_handle_dialog
  Params: { "accept": true }
  期待結果: 削除処理が実行される

Step 5: リダイレクト完了を待ってスナップショットを取得する
  Tool: browser_snapshot
  期待結果:
    - URL が /trips を含む（旅行一覧ページにリダイレクトされている）
    - 削除した旅行のカードが一覧に表示されていない

  ※ リダイレクトが完了していない場合は数秒待ってから再度スナップショットを取得する

Step 6: 削除された旅行に直接アクセスして 404 を確認する
  Tool: browser_navigate
  Params: { "url": "http://localhost:3900/trips/{tripId}" }
  ※ {tripId} は削除した旅行の ID

  Tool: browser_snapshot
  期待結果:
    - 旅行トップページの内容が表示されない（エラー画面、読み込み中のまま、または一覧にリダイレクトされる）
```

### 判定基準

- 削除ボタンクリック後に confirm ダイアログが表示される
- ダイアログ承認後に旅行が削除される
- 削除後に `/trips` にリダイレクトされる
- 一覧から削除された旅行が消えている
- 削除された旅行の URL に直接アクセスしてもデータが表示されない

## 実行時の注意事項

### テスト実行順序

テストケースは番号順に実行することを推奨する。特に以下の依存関係に注意する:

1. TC-002-01 と TC-002-02 は独立して実行可能
2. TC-002-03 は旅行を作成するため、TC-002-04、TC-002-05、TC-002-06 の前提になる
3. TC-002-04 は TC-002-03 で作成した旅行が必要
4. TC-002-05 は TC-002-03 で作成した旅行が必要（編集対象）
5. TC-002-06 は旅行を削除するため、TC-002-05 の後に実行する（削除後は対象旅行が消滅する）
6. TC-002-07 は独立して実行可能（新しい旅行を作成する）

### 推奨実行順

1. TC-002-01（旅行一覧ページの表示）
2. TC-002-02（旅行作成ページの表示）
3. TC-002-03（旅行の新規作成）
4. TC-002-04（旅行トップページの表示 -- TC-002-03 の直後に実施）
5. TC-002-05（旅行情報の編集 -- 作成した旅行に対して実施）
6. TC-002-06（旅行の削除 -- 編集した旅行を削除して一覧に戻る）
7. TC-002-07（メンバー追加した旅行作成 -- 独立したケース）

### ログイン状態について

- 全テストケースはたろう (`taro@example.com`) でログイン済みの状態を前提とする
- ログインセッションが切れた場合は、TC-001-02（シナリオ 001）の手順でログインし直すこと

### 日付入力について

- 日付フィールド（`type="date"`）は `browser_fill_form` を使用して `YYYY-MM-DD` 形式で入力する
- ブラウザによっては日付ピッカーの挙動が異なるため、直接値を設定する方法が確実である

### confirm ダイアログについて

- TC-002-06 の削除確認では、ブラウザの `window.confirm()` が使用される
- `browser_handle_dialog` ツールで承認（accept）または拒否（dismiss）を制御する
- ダイアログが表示される前に `browser_handle_dialog` を準備しておく必要がある場合がある

### browser_snapshot の ref 番号について

- `browser_snapshot` で取得したスナップショットには、各インタラクティブ要素に ref 番号が割り当てられる
- `browser_click` や `browser_type` では、この ref 番号を使って要素を指定する
- 本仕様書では要素の説明テキストで記載しているが、実行時はスナップショットの ref 番号に置き換えること

### エラー発生時の対応

- MCP ツールの呼び出しでエラーが返された場合は、即座にエラー内容を報告する
- 回避策の実行やリトライは行わない（CLAUDE.md の Playwright MCP 使用ルールに準拠）
- 期待結果と異なる場合は、実際の状態をスクリーンショットまたはスナップショットで記録し、差異を報告する
